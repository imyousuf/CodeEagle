# Facescan: Face Detection, Clustering & Recognition

## Overview

Facescan is a standalone CLI tool for detecting faces in personal photo collections, extracting face embeddings, clustering faces by identity, and supporting person labeling and search. It lives at `tools/facescan/` as a standalone Go module, prototyping the face recognition pipeline that will later integrate into CodeEagle's non-code file support (see [non-code-file-support.md](non-code-file-support.md)).

## Architecture

### Standalone Tool

```
tools/facescan/
├── main.go           # CLI entry point, scan/clusters/label/search/suggest/stats commands
├── scanner.go        # OpenCV DNN face detection + SFace embedding extraction
├── cluster.go        # Agglomerative clustering (average linkage) + noise absorption + DBSCAN
├── merge_split.go    # Manual merge/split commands for cluster correction
├── extract.go        # Face thumbnail extraction for visual verification
├── store.go          # FaceDB: JSON-based persistence of faces, embeddings, cluster assignments
├── models.go         # Model download management (auto-downloads on first run)
└── go.mod            # Standalone module: github.com/imyousuf/facescan (gocv v0.31.0)
```

Data stored at `~/.facescan/`:
- `data.json` — face records with embeddings, bounding boxes, cluster assignments, labels
- `models/` — DNN model files (auto-downloaded)
- `faces/` — extracted face thumbnails grouped by cluster (generated by `extract` command)

### Dependencies

```
gocv.io/x/gocv v0.31.0  — Go bindings for OpenCV 4.6.0 (requires CGO + system OpenCV)
```

**System requirement**: `sudo apt install libopencv-dev` (Ubuntu) or equivalent. gocv v0.31.0 is the only version compatible with Ubuntu's packaged OpenCV 4.6.0.

## CLI

```
facescan scan <dir1> [dir2] ...   Detect faces, extract embeddings, cluster
facescan clusters                  Show face clusters with sample images
facescan extract                   Extract face thumbnails grouped by cluster
facescan suggest                   Show merge candidates (similar clusters)
facescan merge <id1> <id2> [id3..] Merge clusters (same person split apart)
facescan split <id> [--sim <f>]    Re-cluster a single cluster at tighter threshold
facescan label <cluster-id> <name> Assign a person name to a cluster
facescan search <name>             Find images containing a named person
facescan stats                     Show scan statistics
```

### Scan Options

| Flag | Default | Description |
|------|---------|-------------|
| `--force` | false | Re-scan all images (ignore previous results) |
| `--threshold` | 0.7 | Face detection confidence threshold |
| `--sim` | 0.30 | Average-linkage similarity threshold for clustering |
| `--min-face` | 40 | Minimum face size in pixels |
| `--max-res` | 1024 | Max image resolution for detection (longest edge) |

## Face Detection Pipeline

### DNN Models

Two models, auto-downloaded on first scan:

| Model | File | Size | Purpose |
|-------|------|------|---------|
| SSD Face Detector (Caffe) | `deploy.prototxt` + `res10_300x300_ssd_iter_140000.caffemodel` | ~10 MB | Face bounding box detection |
| SFace Recognizer (ONNX) | `face_recognition_sface_2021dec.onnx` | ~37 MB | 128-dim L2-normalized face embeddings |

Source: OpenCV Zoo GitHub releases.

### Multi-Scale Detection

The SSD detector uses a fixed 300x300 input, which misses small faces in high-resolution images. Facescan uses multi-scale tiling:

1. **Full image pass** — catches large/medium faces
2. **2x2 tiles with 60px overlap** — each tile is ~half the image, so faces appear ~2x larger to the SSD model
3. **3x3 tiles** (if image >= 600x400) — catches even smaller faces in large group shots

All tile coordinates are mapped back to the full resized image. NMS (non-maximum suppression, IoU threshold 0.3) removes duplicate detections across scales/tiles.

### Embedding Extraction

For each detected face:
1. Crop face region with 20% padding from the resized image
2. Resize crop to 112x112 (SFace input requirement)
3. Create blob: scale=1.0, swap BGR→RGB, no mean subtraction
4. Forward pass through SFace → 128-dim embedding
5. L2-normalize the embedding (making cosine similarity = dot product)

**Note on face alignment**: The current implementation uses simple crop+resize without facial landmark alignment. This is a known limitation — SFace was designed for aligned faces (the nominal same-person threshold of cosine similarity > 0.637 assumes alignment). Without alignment, the effective discrimination is lower, requiring a lower clustering threshold (0.30 vs 0.637). Proper 5-point landmark alignment would improve embedding quality but requires a reliable landmark detector. Haar cascade eye detection was tested and abandoned (unreliable on small crops, made results worse).

## Clustering

### Algorithm: Agglomerative Clustering with Average Linkage

**Why not DBSCAN**: DBSCAN uses transitive (single-linkage) chaining — if face A matches face B, and face B matches face C, then A and C end up in the same cluster even if they look nothing alike. This is catastrophic for family photos where siblings/parent-child have genuinely similar embeddings (measured up to 0.932 cosine similarity between brothers in test data). A single high-similarity outlier pair chains entire families into one cluster.

**Average linkage** considers ALL cross-cluster face pairs when deciding whether to merge. The merge criterion is the mean similarity across all face pairs, not just the best one. This prevents a single outlier pair from dominating the merge decision.

```
Algorithm:
1. Initialize: each face is its own cluster (N clusters)
2. Precompute N×N pairwise cosine similarity matrix
3. Loop:
   a. For every cluster pair (i, j):
      - Check same-photo constraint (reject if any image appears in both)
      - Compute average similarity across all cross-cluster face pairs
   b. Find the pair with highest average similarity
   c. If best average similarity >= threshold (default 0.30): merge them
   d. Otherwise: stop
4. Renumber clusters contiguously; singletons (< minClusterSize=2) become noise (-1)
```

**Same-photo constraint**: A person can only appear once per photo. If two clusters share an image (i.e., one face from photo X is in cluster A, another face from photo X is in cluster B), those clusters cannot be the same person. This constraint is enforced at every merge decision and is the primary guard against merging different people who appear in the same group photos.

**Complexity**: O(N³) where N is the number of faces. Each merge iteration scans all cluster pairs (O(k²) where k is current cluster count), and computing average similarity requires iterating over all cross-cluster pairs. For typical personal photo libraries (<1000 faces), this completes in milliseconds.

### Noise Absorption

After agglomerative clustering, remaining noise faces (singletons below `minClusterSize`) are absorbed into clusters via single-pass best-match:

1. For each noise face, find the cluster with the highest best-match face similarity
2. If best match >= absorption threshold (default: 75% of clustering threshold) AND no same-photo violation: assign to that cluster
3. **Single-pass only** — no iteration, to prevent chaining through absorbed faces

This is critical: iterative absorption creates the same transitive chaining problem we avoided with average linkage. Each absorbed face must be assigned independently based only on the original cluster membership.

### DBSCAN (Split Command Only)

DBSCAN is retained for the `split` command, which re-clusters a single cluster at a tighter threshold (default 0.75). Within a single cluster, the faces are expected to be reasonably similar, so DBSCAN's transitive chaining is less problematic. Same-photo constraint is enforced in both `regionQuery` (neighbor lookup) and during cluster expansion (assignment-level check via `clusterImages` map).

### Threshold Rationale

The default clustering threshold of **0.30** was determined empirically using a ground truth dataset of 88 faces across 8 people and 3 events:

| Threshold | Intra-Person Pairs Caught | Inter-Person False Positives | Notes |
|-----------|---------------------------|------------------------------|-------|
| 0.637 | 13.9% | 0.5% | SFace nominal (with alignment) — too strict without alignment |
| 0.500 | 27.6% | 2.6% | Moderate — still too much noise |
| 0.400 | 48.2% | 14.7% | Catches most pairs but DBSCAN chains badly |
| 0.350 | 58.7% | 24.8% | Good for agglomerative (average linkage resists false positives) |
| 0.300 | 70.7% | 37.6% | **Sweet spot for average linkage** — high coverage, same-photo constraint blocks most false merges |

At 0.30 with average linkage: 11 clusters from 88 faces, 0 noise, 5/8 people have 100% pure clusters. The main remaining issue is fragmentation for people whose appearance varies significantly across events (different lighting, beard, clothing). These fragments can be manually merged using the `merge` command after reviewing thumbnails.

### Test Results

Ground truth: 8 people, 88 faces, 3 family events, 0 ideal noise.

| Person | Faces | Best Cluster Recall | Purity | Notes |
|--------|-------|---------------------|--------|-------|
| Imran | 11 | 91% (10/11) | 100% | 1 face in noise absorbed elsewhere |
| Hamza | 23 | 61% (14/23) | 93% | Fragmented across 3 clusters (cross-event variation) |
| Mahdi | 20 | 40% (8/20) | 80% | Fragmented across 4 clusters (hardest case — brothers look similar) |
| cluster_06 | 6 | 100% | 100% | Single-event, perfect |
| cluster_07 | 6 | 100% | 100% | Single-event, perfect |
| cluster_08 | 6 | 100% | 86% | 1 contamination face |
| Farhana | 13 | 100% | 87% | All faces in 1 cluster, 2 contamination faces |
| cluster_10 | 3 | 100% | 100% | Single-event, perfect |

**Key finding**: The primary quality bottleneck is SFace embedding quality without face alignment, not the clustering algorithm. Same-person face pairs across different events can have cosine similarity as low as 0.005 (essentially random), while family members (brothers, mother-son) can have cross-person similarity up to 0.932. No clustering algorithm can cleanly separate these distributions — the manual merge/split/suggest workflow handles the remaining cases.

## Manual Correction Workflow

The automatic clustering provides a strong starting point. The correction workflow is:

1. **`facescan extract`** — generates face thumbnails grouped by cluster in `~/.facescan/faces/`
2. **Visual inspection** — user opens the folder and checks cluster purity
3. **`facescan suggest`** — shows merge candidates sorted by average similarity, with same-photo overlap detection:
   - "LIKELY same person" — avg similarity >= 0.30, no image overlap
   - "possibly same person" — avg similarity >= 0.25, no image overlap
   - "different people (shared photos)" — image overlap detected (different people in same group photo)
4. **`facescan merge <id1> <id2>`** — merge confirmed same-person clusters
5. **`facescan split <id> [--sim <f>]`** — re-cluster a mixed cluster at a tighter threshold
6. **`facescan label <id> <name>`** — assign person names
7. **`facescan search <name>`** — find all images containing a named person

### Merge

Reassigns all faces from source clusters to the target cluster (first ID). Migrates labels (if target is unlabeled but a source has a label, the source's label is used).

### Split

Re-runs DBSCAN within a single cluster at a tighter threshold (default 0.75). Sub-cluster 1 keeps the original cluster ID; subsequent sub-clusters get new IDs. Faces that become noise get cluster ID -1.

### Suggest

Computes pairwise average and max cosine similarity between all cluster pairs. The same-photo overlap flag is the most reliable signal: if two clusters share an image, they are almost certainly different people (since each person appears only once per photo).

## Data Model

### FaceRecord

```go
type FaceRecord struct {
    ImagePath  string          `json:"image_path"`   // absolute path to source image
    FaceIdx    int             `json:"face_idx"`     // face index within image (0-based)
    BBox       image.Rectangle `json:"bbox"`         // bounding box in original image coordinates
    Confidence float32         `json:"confidence"`   // detection confidence (0-1)
    Embedding  []float32       `json:"embedding"`    // 128-dim L2-normalized SFace embedding
    ClusterID  int             `json:"cluster_id"`   // cluster assignment: -1=noise, 1+=cluster
}
```

### FaceDB

```go
type FaceDB struct {
    Faces  []FaceRecord   `json:"faces"`   // all detected faces
    Labels map[int]string `json:"labels"`  // clusterID → person name
}
```

Persisted as JSON at `~/.facescan/data.json`. Simple file-based storage appropriate for the prototype; the CodeEagle integration will use BadgerDB (see below).

## Integration Path to CodeEagle

When integrating into CodeEagle's non-code file support pipeline:

1. **Package location**: `internal/faces/` (behind `//go:build faces` tag for optional OpenCV dependency)
2. **Storage**: BadgerDB at `.CodeEagle/face.db/` with key prefixes: `face:emb:`, `face:box:`, `face:cluster:`, `face:label:`, `face:person:`
3. **Graph integration**: labeled clusters become `NodePerson` nodes connected to `NodeDocument` images via `EdgeAppearsIn` edges
4. **CLI**: `codeeagle faces scan|clusters|label|search|merge|split|suggest` subcommands
5. **Clustering algorithm**: carry over agglomerative average-linkage (not DBSCAN) — validated against real family photos
6. **Same-photo constraint**: carry over unchanged — fundamental to family photo correctness
7. **Object detection**: add YOLOv8n alongside face detection, producing `NodeTopic` nodes for detected objects

See [non-code-file-support.md](non-code-file-support.md) for the full integration design.

## Known Limitations

1. **No face alignment** — SFace embeddings are extracted from simple crop+resize without landmark-based alignment. This degrades embedding quality, especially across different lighting/angle/events. The nominal SFace threshold of 0.637 drops to an effective 0.30 for average-linkage clustering.

2. **Family resemblance** — siblings and parent-child pairs can have cosine similarity up to 0.932 (measured between brothers in test data). No embedding model or clustering algorithm can reliably separate people who genuinely look alike. The same-photo constraint prevents merging people who appear in the same photo, but cross-event matching of look-alike family members requires manual correction.

3. **Cross-event variation** — a person's appearance can vary significantly across events (lighting, facial hair, aging, clothing). Same-person face pairs across events can have cosine similarity as low as 0.005. Average-linkage clustering handles this better than DBSCAN (no transitive chaining), but some fragmentation is expected.

4. **Scalability** — agglomerative clustering is O(N³) and the similarity matrix is O(N²) memory. For >1000 faces, approximate methods (HNSW for neighbor queries, mini-batch clustering) would be needed. The current brute-force approach is adequate for personal photo libraries up to ~1000 faces.

5. **gocv compatibility** — locked to gocv v0.31.0 due to Ubuntu's OpenCV 4.6.0 packaging. Newer gocv versions may require building OpenCV from source.
